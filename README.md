## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

私は以前から、何かを思いついたとき時や、何か始めたいと思った時などに、ノートやメモ帳アプリへ、その時の考えを書き出していました。
自分の頭の中をノートなどに書き出す作業は、「ジャーナリング」と呼ばれています。
ジャーナリングを通じて、自分の考えを書き出してみると頭の中が整理されて、「では、まず何をしてみようか」と考えられるようになります。

実際に、

- 行きたかった場所へ旅行に行く
- スカイダイビングに挑戦する
- キャンプを体験する
- TOEICの学習を始める

など、書き出したことをキッカケに、行動に繋がった経験が何度もありました。

そのため、思い付いたことを書き出したり、やりたいことを書き出す作業には、私は大きな価値があると感じています。

しかし、同時に一つ課題もありました。
それは**「続けることが難しい」**ということです。

やる気がある日は、ノートを開いて、たくさん考えます。
しかし、数日経つと、「今日は疲れたからいいかな」「時間がないから明日にしよう」となり、気付けばノートを開かなくなってしまいます。
2～3日は続いても、習慣化が難しく、1週間、1か月と時間がたつと、継続することができませんでした。

振り返ると、私はジャーナリングに対して、高いハードルを設定していました。
ジャーナリングに取り組もうとしたとき、無意識のうちに「たくさん書き出さなければならない」「たくさん考えなければならない」と思い込んでいたため、書き始める事、続ける事が負担になっていました。

そんな中、ある学習アプリを使った経験がありました。

そのアプリには、

- 経験値
- レベルアップ
- 継続日数の表示

といった、ゲーム要素がありました。

さらに、1回の学習が1分程度で終わるように設計されていた為、学習し始めるハードルが非常に低くなっていました。
電車の待ち時間やレンジでチンする時の時間など、ちょっとした空き時間があれば、アプリを開けます。

この学習アプリは、「習慣化するのが難しい」と感じていた私の課題を解決してくれました。
気付けば、毎日アプリを開くことが習慣となり、継続できるようになっていました。

その経験から、

「ジャーナリングにも同じ仕組みを取り入れれば、続けられるのではないか」

と考えるようになりました。

### 2-2. なぜそれが「気になった」のか

私には、「成長したい」「新しいことに挑戦し、体験して思い出を増やしたい」「できなかった事ができるようにしたい」といった価値観や思いがあります。
その一方で、「行動を始めるまでに時間がかかる」「やりたいと思っていても、後回しにしてしまう」「マイペースで、自分の時間を大切にしたい」といった、怠惰でいたいという一面もあります。

実際、TOEIC学習も長い間「やりたい」と思っていましたが、なかなか始められませんでした。

しかしある時、ノートに「英語を読めるようになりたい」「TOEICを受けたい」と書き出したことで意識する機会が増えました。すると、本屋でTOEICの参考書が目に留まったり、関連動画を見るようになったりして、少しずつ行動につながりました。

最終的には学習アプリを使い始め、継続的に勉強するようになりました。

この経験から、**人は「やりたいこと」を言語化することで、意識することが多くなり、行動しやすくなる**と感じています。

逆に、頭の中だけで考えていることは簡単に忘れてしまいます。
昨日考えていたことも、数日後には思い出せなくなることがあります。
だからこそ、**自分のやりたいことや感情を言語化して外に残すこと**には大きな価値があると思っています。

現在はAIやインターネットの発達によって、やり方や知識は簡単に手に入ります。

しかし、

- 自分は何をやりたいのか
- 何に興味があるのか
- どんな人生を送りたいのか

といった主観的な気持ちは、自分自身にしか分かりません。

だから私は、自分の「やりたい」という感情を大切にしたいと思っています。

そのために、毎日1つだけでもやりたいことを書き出す。
そして、無理なく継続できる仕組みを作る。

そんなアプリがあれば、自分自身だけでなく、同じように「やりたいことはあるけれど続かない」と悩んでいる人の役に立てるのではないかと考えました。

## 8. このアプリの懸念点とその対策

このアプリが**うまくいかない可能性**と、
それに対して**どう向き合おうとしているか**を書いてください。

### 懸念点①

- 何が問題になりそうか：
　もっと、やりたいことを登録したいと感じて、他アプリを使うことになる

- なぜそう思うか：
　1日1個しか登録できない仕様になっているため、やる気のある人はじれったく感じてしまうと思われる

- そのために考えている対策：
　トップ画面で、このアプリの思想を分かってもらう文言やイラストを表示して理解してもらう
　使ってもらうアプリの対象者の像（ターゲット）をハッキリと伝える
　（やりたいことリストの習慣化に失敗したことがある人、なまけものの自負がある人）
  すでにやりたいことリストをバリバリ継続できている人は、使う意味がありません等、使わない人の想定も正直に書く
　＜本アプリの思想＞
　やる気のあるときだけ、たくさんやりたいことを書いても、やる気が出ないこともあるので、継続が難しい。
　ハードルを低く設定して、あえて制限を付けることで、小さく始めて長く付き合っていくこと、習慣化することを想定している。
　自分のやりたいことと向き合うのは1日だけじゃ効果は薄い。
　長く続けてこそ、やりたいことと向き合えて、たくさんのやりたいを忘れることなく実現できる。

### 観点

- ユーザーはなぜ使い続ける？

ハードルを徹底的に低くすることで、やらない理由がない。
レベルアップ要素を入れることで、成長が可視化される。

- 続けることで何が変わる？

やりたいことリストを書くだけでなく、
実際にやりたいことが達成されていって行動が変わっていくのを体感する
やりたいことリストを共有できる機能を付けることで、人との繋がりも生まれる
人の繋がりを活発にするために、アイコン画像ややりたいことのイメージ画像なども保存できるようにする

- データが溜まると何ができる？

ユーザーが書いたやりたいことリストをもとに
ランダムに表示されるやりたいことの種類が増えていく
心理テストを実施→心理テスト結果が同じユーザーがよく登録しているやりたいことをユーザーに提案する

- 他の人と関わる余地はある？

やりたいことリストの共有やコメント機能、いいね機能で
どのようにやりたいを実践していくかなど
ユーザー同士がアドバイスしながら交流できる可能性がある

### 今考えている発展案

- UI・体験の改善：レベルアップする時に、ポケモンの経験値やProgateのように、バーで成長している感覚を可視化する
- 機能以外の工夫：自ら毎日このアプリを使って、SNSで使い心地ややりたいことリストを公開する。自分自身が、このアプリを通して出会ったやりたいことを実現することでアプリの良さを広める

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- フレームワーク：Ruby on Rails
- DB：PostgreSQL
- デプロイ先：Render + Neon + Cloudinary
- 使用予定ライブラリ：
 - Devise：ユーザー登録、ログイン関連
 - Pundit：権限関連
 - Kaminari：ページネーション or Pagy(Kaminariより高速)
 - Ransack：検索機能(カテゴリ・達成状況等)
 - Stimulus：レベルアップ演出(経験値バー等)
 - GSAP：高度なアニメーション機能(紙吹雪、ボタンが弾む等)

---

### 10-2. 技術選定の理由

#### 言語/フレームワーク

| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| Ruby on Rails | 採用 | ① Railsガイドなど、日本語の記事や書籍が多くあり、キャッチアップしやすい。また、既にRailsの学習を進めているため、学習コストが低い。<br><br>② 「設定より規約」という設計思想により、命名規則やフォルダ構成などRailsのルールに従うことで、本質的でない設定に悩む時間が減り、アプリの仕様や機能など本質的な課題に集中できる。<br><br>③ CRUD処理や認証機能など、Webサービス開発で必要となる機能が標準的に揃っており、今回開発するアプリを効率的に実装できる。 |
| Vue | 不採用 | 学習コストは低いが、就活先の求人がReactの方が多かった |
| Svelte | 不採用 | 情報が少なく、詰まったときに解決しにくい |

#### PostgreSQL：
1. RailsはRDBを扱う事を前提としていて、Railsと相性がいい。
  1-1. Railsの新機能はまずPostgreSQLで検証される
2. PostgreSQL専用のデータベースサービスであるNeonを採用するため
3. 初期のInstagramなど、本番で運用される実績が多く、メジャーなDBのため

#### Render + Neon：
1. 現時点で無料枠があり、初期段階では運用コストがかからない
2. 一度設定すれば、GitHubへpushで自動デプロイされるため、すぐに実装を本番環境で試すことができる
3. RenderとNeonの接続設定が比較的に容易で、Neonが発行した接続文字列をRenderに環境変数で設定するだけで使える

- 今回チャレンジしたい点
 - 習慣化がメインテーマの為、継続することによる報酬として、レベルアップのアニメーションを、もっとレベルアップしたくなるような体験にしたい
 - ポイントシステムで、やりたいこと登録が100pt消費する、ポイントが足りないと登録はできない、ログインで1日1回100pt付与、ランダム表示は1回につき10個まで等、制限の機能をRailsのロジックで実現すること

- 不安な点
 - Stimulus等のRailsでの非同期処理の理解が曖昧なため、実装時にキャッチアップが必要
 - ページをスムーズに動かすためのパフォーマンス向上のための知見があまりない

---

### 10-3.キャッチアップ不足の懸念

1. 調査範囲が不明確で、キャッチアップに時間がかかる可能性がある1
- StimulusやRailsでの非同期処理について理解が浅く、どのような仕組みで動いているかを整理し、今回の機能実装にはどういった設計で取り入れるかを考える必要がある。
- 「何を調べれば実装できるのか」という調査範囲自体が曖昧なため、情報収集や試行錯誤に時間がかかる可能性がある。

2. Issueの工数見積もりが難しい
- 現時点では理解できている部分と理解不足な部分の境界が明確ではない。
- 調査時間や実装難易度を正確に見積もることが難しく、結果としてIssueごとの工数や開発期間の予測精度が下がる可能性がある。

3. 設計や実装のベストプラクティスの判断が難しい
- レベルアップアニメーションやポイントシステムなどは実現方法が複数考えられる。
- 保守しやすい設計や実装が分からず、実装方法の比較・検討に時間を要する可能性がある。

4. パフォーマンス改善の方法やボトルネックとなっている部分の調査方法などが分からない
- ページ表示やアニメーションを快適にするための知識が少なく、どこがボトルネックになっているかを判断するのが難しい。
- パフォーマンス改善には様々な手法があるため、どの改善策を優先すべきか調査が必要になる可能性がある。

5. ロジックの責務設計に悩む可能性がある
- ポイント付与・消費や利用制限などの機能では、Controller・Model・Serviceなど、どこに処理を書くべきか判断が難しい。
- 設計方針を誤ると、後から機能追加や保守がしづらくなる可能性がある。
