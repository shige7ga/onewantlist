# わんわんとリスト（One Want List）

## 1. サービス概要

- どんなサービスか

  やりたいことを「1日1個だけ」書き貯めていく、習慣化に特化したやりたいことリストアプリ。<br>
  1日1個までだから「ちゃんと書かなきゃ」と気負わず、サクッと登録できて、思い付かない日はアプリがランダムで提案してくれる。<br>
  続けるほどレベルアップして、ゲーム感覚で「やりたい気持ちと向き合う習慣」が自然と身につく。

- 誰の、どんな課題を、どう解決するか

  生活が安定し、ルーティン化した日々に「このままでいいのか」と焦る30代社会人。<br>
  日常で「やってみたい」と感じる瞬間はあるが、意識し続ける仕組みがないため、その気持ちが行動に変わる前に数日で消えていく。課題は「やりたいことがない」ことではなく、「やりたい気持ちを忘れたまま日々が過ぎ、後から『やっておけばよかった』と後悔してしまう」こと。<br>
  本アプリは「1日1個だけ」とあえて制限をかけて記録のハードルを極限まで下げ、思い付かない日はランダム提案で補い、レベルアップで継続を報酬化することで、意志力ゼロでも気持ちを拾い続けられる仕組みを提供する。

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

私は以前から、思いついたことや挑戦したいこと等を「やりたいことリスト」としてノートやメモアプリに書き出していた。実際に、旅行やスカイダイビング、キャンプ、TOEIC学習等、書き出したことをきっかけに行動へ移せた経験が何度もある。頭の中を整理し、「まず何をしようか」と考えるきっかけになるため、やりたいことを書き出す事に大きな価値を感じている。

しかし、一つ大きな課題がある。それは「継続が難しい」こと。最初は意欲的に取り組めても、「今日は疲れた」「また明日でいいか」と次第に書かなくなり、習慣化できなかった。振り返ると「たくさん書かなければならない」と無意識に考え、始めるハードルを自分で高くしていたことが原因だと感じる。

そんな時、1回の学習が1分ほどで終わる学習アプリを使う機会があった。電車の待ち時間やちょっとした空き時間でも気軽に取り組めるため、「始めること」への負担がほとんどなく、気付けば毎日続けていた。また、継続日数やレベルアップなど、成長を可視化するゲーム要素もあり、継続するモチベーションに繋がった。

この経験から、「やりたいことリストにも、取り組むハードルを下げ、成長を実感しながら続けられる仕組みを取り入れれば、習慣化しやすくなるのではないか」と考えた。

---

### 2-2. なぜそれが「気になった」のか

私には「成長したい」「新しいことに挑戦し、思い出を増やしたい」という価値観がある一方で、「行動を始めるまでに時間がかかる」「やりたいと思っても後回しにしてしまう」という一面もある。

実際、TOEIC学習も長らく「やりたい」と思いながら始められていなかった。そんな中、ノートに「英語を読めるようになりたい」と書き出した事をきっかけに意識する機会が増え、参考書が目に留まったり関連動画を見るようになったりと、少しずつ行動につながった。最終的には学習アプリで継続的に勉強するようになった。

この経験から、人は「やりたいこと」を言語化することで意識するようになり、行動しやすくなると実感している。逆に、頭の中だけで考えていることは数日で忘れてしまいがち。だからこそ、自分の気持ちを言語化することに、大きな価値があると確信している。また、AIやインターネットの発達でやり方や知識は簡単に手に入る一方、「自分は何をやりたいのか」「どんな人生を送りたいのか」といった主観的な気持ちは、自分自身にしか分からないため、やりたいことを明確にすることの重要性は増していると感じる。

そこで、毎日1つだけでもやりたいことを書き出し、無理なく継続できる仕組みを作れば、自分自身だけでなく、同じように悩む人の役に立てるのではないかと考えた。

## 3. 課題の整理

### 3-1. 表に見えている困りごと

やりたいことリストを書き始めても継続が難しく、習慣化できないため、やりたいことを忘れたり行動につながらなかったりする。

---

### 3-2. 本当に解決したい課題は何か

#### なぜそれが困るのか？

- 「やりたい」と思っても、書き留めたり意識し続けたりする習慣が続かないため、その気持ちは時間とともに薄れ、忘れたり後回しにしてしまう。

- 気持ちが薄れることで行動に移すタイミングを逃し、新しい挑戦や経験が増えないため、自分の価値観や可能性を広げる機会を失う。

- そうした機会損失が積み重なることで、人生が今までの延長線上になりやすく、「あの時やっておけばよかった」と後悔するような、思い出や経験の少ない人生になってしまう。

#### 本当に解決したい課題は？

人は「やりたい」と思っても、それを意識し続ける習慣を保てないため、気持ちが時間とともに薄れ、行動に移すタイミングを逃す。その結果、新しい挑戦や経験の機会を逃し続け、代わり映えのない日々を送り、後々「やっておけばよかった」という後悔を抱えてしまう。

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

- 年齢・性別：
  - 定義：34歳・独身・一人暮らしの男性。関東のメーカー勤務の営業で、仕事はほぼメール対応の内勤。友人は少なめで、遊ぶといっても飲みに行くくらい。
  - 理由：30代半ばは、仕事や生活が安定してルーティンが固まる一方、同年代の友人は結婚していき、飲みに誘える相手も減っていく年代。「このまま同じ毎日が続くのかな…」という焦りから、「何か新しいことをやりたい」という気持ちが生まれやすい。動画で見た料理や会話に出た旅行先など、「やってみたいかも」と思う瞬間は日常にある。しかし記録する習慣がないため数日で忘れ、行動につながる前に消えてしまう。この課題が深刻なのが、変化の少ないこの年代だと考えた。

- 生活・状況：
  - 定義：朝9時から18時のフルタイム勤務で、通勤は電車で片道1時間弱。昼はいつもと同じ飲食店で同じ定食を食べる。平日夜は疲れて動画を見て終わり、休日は昼まで寝てしまう。以前「やりたいことリスト100」に挑戦したが、30個あたりで手が止まり、手帳は3か月以上開いていない。
  - 理由：「100個書くぞ」と意気込んだせいでリストへのハードルが上がり、疲れた平日夜には手が伸びず、挫折した経験を持つ。「たくさん書かなきゃ」という思い込みが、続かない原因になっている。一方で、片道1時間の通勤のようなスキマ時間は毎日ある。この「毎日の小さな時間」で無理なく記録を習慣化できる仕組みを、本アプリで提供する。

---

### 4-2. 自分とユーザーの距離

  想定ユーザーは、ほぼ自分自身。やりたいことの言語化に価値を感じながらも継続できない、という悩みをまさに今の自分が抱えている。そのため、課題の自分事として考えて開発でき、「この機能で本当に続けられるか」を自分自身で検証できる。

---

### 4-3. 実際に使われる可能性について

- まず最初に使ってくれそうな人は誰か？

  やりたいことリストを書いた経験があり、継続できずに挫折したことがある人。ノートやメモアプリで一度は試している時点で「書き出すことへの価値」はすでに理解しており、本アプリの説明コストが低い。課題は「始めること」ではなく「続けること」にあるため、継続の仕組みを軸とする本アプリと最も相性が良い層と考えられる。

- その人は、どんな場面でこのアプリを思い出すか？

  「毎日同じ生活の繰り返しだな」と感じたとき。何か始めたい、刺激が欲しいと思いながらも行動に移せないとき。SNSで充実した投稿を見て、漠然と「自分も何かしたい」と感じたとき。いずれも、想定ユーザーの「このまま同じ毎日が続くのかな」という焦りが表面化する瞬間であり、そこが本アプリの入口になる。

- 今の生活の中で、代わりに使っているものは何か？

  iPhoneのメモ帳、Notionなど。ただしこれらは「書く場所」を提供するのみで、続けるための仕組みは持たない。実際、想定ユーザー(=自分自身)はこれらの手段で、継続できていない。

- それを置き換えてまで使う理由はあるか？

  代替手段との違いは、「記録する場所」ではなく「記録し続けられる仕組み」を提供する点。

  1. 1日1個だけの登録という低いハードルにより、「たくさん書かなきゃ」という思い込みを仕組みで排除し、習慣化しやすくする
  2. ランダム提案により、やりたいことが思い付かない日でも「これ、やってみたいかも！」と考えるきっかけができ、やりたいことを意識する習慣が途切れない
  3. レベルアップ機能により継続が見える化され、続けること自体がモチベーションになる

  既存ツールが解決していない「継続」に特化していることが、置き換える理由である。


## 4-4. ユーザーがサービスを導入して利用するまでのイメージ

- サービスを知る・思い出すきっかけ

  やりたいことリストをノートやメモアプリで書いた経験はあるが、続いたことがない。あるいは「始めたい」と思いつつ、ハードルを高く設定しすぎて手を付けられずにいる。そんな人が **怠け者のための、ハードルを極限まで下げた「やりたいことリスト」** というキャッチフレーズを目にしたとき、「これなら続きそう」「時間がかからなそう」と感じることが、利用開始のきっかけになる。

- 最初の一歩

  「まず触れる → 良さが分かる → 登録する」の順で、導入のハードルを最小化する。

  1. 画面を開いた瞬間、経験値を獲得してレベルアップ(Lv1→Lv2)する演出が流れる

      行動心理学の「エンダウド・プログレス効果(※)」に基づく設計。ゼロからのスタートではなく「すでに進み始めている」状態を作ることで、継続への慣性を最初の数秒で生む。また、「画面を見に来たという小さな行動も、価値ある経験の一つであり、継続の第一歩である」という考えを、言葉ではなく体験で伝える

      ※ Nunes & Drèze(2006)の実験。無料洗車のスタンプカードで「8個必要・0個スタート」と「10個必要・2個押し済みスタート」を比較したところ、必要な実労力は同じにもかかわらず、達成率は19%対34%と後者が約2倍だった。

  2. ログイン不要のまま「やりたいことリスト」を体験できる(localStorageで一時保存)

  3. 2回目以降のやりたいこと登録や他機能を使いたい場合はユーザー登録へ誘導。ユーザー登録でも経験値獲得&レベルアップとして演出す

- 利用を続ける理由

  1. 1日のやりたいことの登録回数の制限:

      登録は1日1個まで。あえて制限することで「たくさん書かなきゃ」という挫折の原因を排除し、「1個で充分」に発想を反転させる。

  2. ランダム提案(続けられる仕掛け):

      やりたいことが思い付かない日でも、アプリを開けば「今日の提案」が届く。考えることが億劫にならず、やりたいことを意識する習慣が途切れない

  3. レベルアップ機能(続けたくなる仕掛け):

      ログイン、やりたいことの登録、連続ログインなど、あらゆる行動で経験値が貯まり、レベルとして継続が見える化される。経験値は減ることがなく、「続けてきた証」として積み上がる

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

- iPhoneメモ帳アプリ
  - 機能:
    - やりたいことを思いついた瞬間に、すぐ書き留められる(起動が速く、操作もシンプル)
    - 件数の制限なく自由に書ける
    - チェックリスト形式で、達成したものにチェックを付けられる
    - フォルダ分けやピン留めで整理できる
    - iCloudで複数端末から同じメモを見られる

- Notion
  - URL：https://www.notion.com/ja
  - 機能:
    - やりたいことを自由に追加できる
    - 件数に制限なく、メモや詳細も追記できる
    - カテゴリや優先順位なども付けることができる（タグを付けられる）
    - 日付管理で期限を設定できる

- 1年でやりたい100のコト NOLTY
  - URL：https://nolty.jp/100nokoto/
  - 機能:
    - 自分が1年以内にやりたいことを最大100個まで登録できる
    - 自由入力で好きな内容を書ける
    - あらかじめ用意された「やりたいことリスト」から検索できる
    - 「やりたいことガチャ」でランダムに候補を表示
    - 自分では思いつかなかった目標や体験に出会える
    - 登録したやりたいことの順番を変更できる
    - 入力内容をブラウザに保存できる
    - 100個のリストを画像として出力できる

---

### 5-2. それでも自分が作りたい理由

  調査した既存サービスは、それぞれ次の課題がある。

  - iPhoneメモ帳：すぐ書ける手軽さはあるが、開く理由も続ける仕掛けもなく意志力任せ。書いたメモも他のメモに埋もれ、見返されなくなる。

  - Notion：多機能ゆえに始めるまでの設定・学習コストが高い。書く枠が無制限で「作り込まなきゃ」というハードルも生まれ、習慣化が得意な人向け。

  - NOLTY 100のコト：ガチャで候補に出会える点は優れているが、「一括で100個」方式で継続の仕組みがなく、リストを作ったら終わりの単発体験になりやすい。

  調査したサービスでは、「書く場所」(メモ帳・Notion)や「出会うきっかけ」(NOLTY)は提供できていても、やりたいことと向き合い続けるための「やりたいことリストを書き続ける仕組み」がない。続くかどうかは全て本人の意志力次第であり、怠け癖があって先送りにしがちな人にとっては脱落する構造になっている。意志力に頼らず続けられる、習慣化に特化したやりたいことリストは、あまり無い。

  そのため、自分で習慣化に特化したやりたいことリストを作りたい。やりたいことを継続的に言語化できる仕組みによって、やりたいことを忘れない人生、「常にやりたいことがある、彩りのある生活」を送る人を増やしたい。

### 5-3. 差別化ポイント

  日々、ふと生まれる「やってみたいな」という気持ちを大切にしたい。そのやりたいを言語化して記録すれば実現に近づくことも知っている。でも、記録し続けるのは難しいと感じている人が、ハードルを下げることで、やりたいという感情と向き合い、言語化する習慣をつくるアプリ。

- 差別化ポイントを一言で表すと

  あえて登録を1日1個に制限することで「たくさん書かなきゃ」を仕組みで禁止した、怠け者・意識低い系のための習慣化特化型「やりたいことリスト」

## 6. このサービスで提供したい価値

### 6-1. ユーザーの変化

このサービスを使うことで、ユーザーはどう変化しますか？

- 行動の変化：
  スキマ時間にアプリを開き、1日1個やりたいことを登録することが日課になる。「たくさん書かなきゃ」というプレッシャーから解放され、書くこと自体が苦でなくなる。

- 気持ちの変化：
  日常の中で「これやってみたいかも」という気持ちに敏感になり、忘れる前に言語化するのが当たり前になる。ランダム提案を通じて興味の幅も広がる。

- 考え方の変化：
  積み上がった経験値とリストが「自分はやりたいことを持ち続けている」という実感になり、なんとなく流される日々から、自分の気持ちを起点に予定を立て、行動する生活へ変わる。

### 6-2. 価値を一言で表すと

  仕事に疲れて帰宅後は何もできない怠け者でも、「1日1個だけ」という低いハードルによってやりたいことを言語化する習慣が身に付き、日々のささいな「やってみたい」を逃さず、自分起点で思い出を作っていけるようになるサービス。

## 7. このアプリで実現すること

### MVPで作る機能

- コア機能：やりたいこと登録（1日1回制限付）
  - 自由入力によるやりたいこと登録
  - ランダム提案機能（思い付かない日でも登録を止めないための補助機能）
    - アプリ側で用意したやりたいこと候補をランダム表示し、ユーザーが気に入ったものをmyリストへ登録できる
      - 気になるやりたいこと候補があったら、内容を自分の言葉に編集して登録できる（編集せず、そのまま登録でも良い）
      - ランダム表示セッション1回につき、やりたいことの表示は10回まで(※登録制限とは別の制限。無限にランダム表示だけする使い方を防ぐため)
      - 表示するだけ(1日10回ランダム表示)でも経験値は付与される(「やりたいことを探した」ことも経験の一つと捉える)

  - 登録数のコントロール
    - やりたいこと登録アクションは1日1回まで(自由登録・ランダム表示どちらか1回)
    - AM0:00に制限解除

  - ゲストユーザー(未ログイン)機能
    - 自由入力での登録:1回だけ体験できる
    - ランダム表示:1セッション(10回表示)だけ体験でき、気に入ったものがあれば1件登録できる(登録は任意)
    - いずれの操作でも経験値が付与される(初回トップ画面表示でのLv1→2演出を含む)
    - データ(リスト最大2件・経験値・Lv)はlocalStorageに保存し、ユーザー登録時にDBへ引き継ぐ
    - 体験を使い切った後の操作時に、ユーザー登録画面へ誘導する

- 認証・導入
  - ユーザー登録・ログイン機能
  - ゲスト利用機能:ログイン不要で「ランダム提案の閲覧」「やりたいことリストへの登録」のみ体験可能(localStorageに一時保存。他機能はログイン後に開放)
  - ゲストデータ引き継ぎ機能:ユーザー登録時、ゲスト時に保存したリストをDBへ移行する

- myやりたいことリスト管理
  - リスト一覧・詳細表示機能
  - 編集・削除機能
  - 達成ステータス管理機能(未着手/着手/達成)
  - 期限設定機能(任意。「いつまでにやるか」を設定できる)

- 継続の見える化機能
  - 経験値・レベルアップ機能(ログイン、登録、ランダム表示で経験値付与)
  - 初回演出:初めてトップ画面を表示でLv1→2、ユーザー登録でLv2→3(エンダウド・プログレス効果)
  - 連続ログイン日数・連続登録日数の表示

### 本リリースで作る機能

- 深掘り機能
  - やりたいこと1件ごとの深掘り機能(テンプレ質問:なぜやりたいか/必要なものは/今できることは+自由テーマ設定)
  - 深掘りレベル機能(※やりたいこと1件に対して、一つ一つレベルを設定)

- 登録体験の拡張
  - カテゴリ別ランダム提案機能
  - タグ別ランダム提案機能
  - 自由入力時に、カテゴリ・タグを設定できる

- 継続の見える化機能の拡張
  - 経験値が上がるタイミングや行動を拡張（連続ログイン、連続登録など）

- コミュニティ機能
  - 他ユーザーのやりたいことリスト閲覧
  - いいね機能
  - コメント機能
  - ユーザーが登録したやりたいことが、ランダム提案のデータベースへ還元される機能
  - やりたいことリストの公開機能（やりたいこと毎に設定可能。デフォルトは非公開）

- 継続支援
  - プッシュ通知機能(リマインド。アプリを開かない日にも「思い出すきっかけ」を届ける)

### 開発タスク(機能ではないが、必要な作業)
  - ランダム提案の初期シードデータ作成(数百件のやりたいこと。本リリースのカテゴリ機能を見越してタグ・カテゴリを付与した状態で用意)
  - 経験値・レベル設計表の作成(各行動の獲得経験値、Lvごとの必要経験値のバランス設計)

## 8. このアプリの懸念点とその対策

### 懸念点

  - 何が問題になりそうか：

    「もっと登録したい」と感じたユーザーが、制限のないメモ帳等へ移ってしまう

  - なぜそう思うか：

    登録は基本1日1個の仕様のため、やる気が高い時期のユーザーには、じれったく感じられる可能性がある

### そのために考えている対策

  1. トップ画面でアプリの思想を明示する

      「やる気がある日だけたくさん書いても、続かなければ忘れてしまう。あえて制限を設けることで、小さく長く続け、やりたいことと向き合い続ける習慣をつくる」

  2. ターゲットを正直に伝える

      - 対象：やりたいことリストの習慣化に挫折した人、怠け者の自負がある人
      - 対象外：すでにリストをバリバリ継続できている人(本アプリは不要と明記)

  3. 仕組み面の逃げ道:続けるほど登録できる数が増える設計

      - 登録に必要なポイントは、毎日のログインやレベルアップ・バッジ獲得で貯まっていく
      - 使わなかった分は翌日以降に持ち越せるため、継続しているユーザーほどポイントの残高が増えていく

## 9. 今後の展開・発展の方向性

### 観点

- ユーザーはなぜ使い続ける？

  ハードルを徹底的に下げることで「やらない理由」を極限まで減らし、レベルアップ要素で成長を可視化しているため、続けること自体がモチベーションになる。

- 続けることで何が変わる？

  リストを書くだけでなく、達成ステータスや期限設定を通じて、やりたいことが実際に達成されていき、行動が変わっていくのを体感できる。

- データが溜まると何ができる？

  ユーザーが登録したやりたいことがデータベースに還元され、ランダム提案の種類が自動的に増え続ける。さらに蓄積データをもとにしたレコメンド(傾向の近いユーザーがよく登録しているやりたいことの提案)へ発展できる。

- 他の人と関わる余地はある？

  リスト共有・いいね・コメント機能を起点に、「どうやって実現したか」をユーザー同士がアドバイスし合う交流の場へ育てられる。

### 今考えている発展案

- 機能追加の方向性
  - AI深掘り支援：AIのAPIを活用し、やりたいことへの深掘り(なぜやりたいか、最初の一歩は何か)を対話形式で支援する
  - レコメンド機能：蓄積データをもとに、傾向の近いユーザーがよく登録しているやりたいことを提案する
  - 振り返り機能：「今年登録した52件、達成12件」のような年間・月間まとめを自動生成し、継続の証を実感できるようにする
  - 通知の進化：「3日前に登録した『キャンプ』、週末どうですか?」のように、登録内容と連動したリマインドで「忘れない仕組み」を強化する
  - 交流の活発化：アイコン画像ややりたいことのイメージ画像を登録できるようにし、コミュニティ機能を盛り上げる
  - 登録機能のさらなる拡張
    - やりたいことのイメージとして、画像も登録できる
    - やりたいことに対して、達成時や挑戦した時の画像を投稿できる
  - ゲーミフィケーション機能の追加
    - バッジ機能
      - バッジの内容案
        - 連続系:7日/15日/30日/100日連続登録、連続ログイン7日・30日
        - 累計系:累計登録10件・50件・100件、累計ログイン30日・100日・365日
    - コレクション・演出
      - イラストコレクション機能(登録ごとにピースが1枚ずつ埋まり、イラストが完成。イラスト難易度に応じて必要ピース数が増加)

- UI・体験の改善
  - レベルアップ時に経験値バーが伸びる演出(Progate等を参考)を入れ、成長している感覚を毎回の操作で可視化する
  - 登録ボタンを押したときに、ボタンが派手に動く機能
  - Lv10など、記念すべき数字の時は、花吹雪やポップアップなど、派手な演出を入れる

- 機能以外の工夫
  - 開発者自身が毎日アプリを使い、SNSで使い心地とやりたいことリストを公開する
  - アプリを通じて出会ったやりたいことを自ら実現していく姿を発信することが、そのままアプリの価値の証明になる

## 10. 技術スタック

### 10-1. 使用予定の技術

- 言語 / フレームワーク：Ruby 3.4系 / Ruby on Rails 8系
- DB：PostgreSQL 18系
- デプロイ先：Render + Neon
- 環境構築：Docker（環境構築を再現可能にし、本番環境との差異を減らす）
- 使用予定ライブラリ：
  - Devise：ユーザー登録、ログイン関連
  - Pundit：権限関連
  - Pagy：ページネーション（軽量・高速のため。実装時にKaminariと比較検討）
  - Ransack：検索機能(カテゴリ・達成状況等)

### 10-2. 技術選定の理由

| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| Ruby / Ruby on Rails | 採用 | Railsガイドをはじめとした日本語の情報が豊富で、既に学習経験があるため、学習コストを抑えながら開発を進められる。<br>また、「設定より規約」の思想により設定で迷う時間が少なく、CRUDや認証機能などの標準機能を活用して、アプリの機能開発に集中できると考え採用した。 |
| PHP / Laravel | 不採用 | Webアプリ開発に適したフレームワークだが、Laravelの学習経験がなく、新たな学習コストが発生する。既に学習を進めているRailsの方が効率よく開発できるため、不採用とした。 |
| PostgreSQL | 採用 | 本番環境で利用するNeonがPostgreSQL専用のサービスであるため採用した。RailsはPostgreSQLとの相性が良く、大規模サービスでの採用実績も豊富で信頼性が高い。 |
| MySQL | 不採用 | 代表的なRDBだが、Neonを利用する本構成ではPostgreSQL一択となるため不採用とした。 |
| Render / Neon | 採用 | どちらも無料枠があり初期コストを抑えられる。GitHubへのpushで自動デプロイされ、NeonとRenderは接続文字列の設定のみで連携できるため、構築・運用が容易である。 |
| Heroku | 不採用 | 無料プランが廃止されており、初期費用を抑えたい今回の要件に合わないため、不採用とした。 |
| AWS | 不採用 | 自由度が高く、本番環境で広く利用されている。しかし、多種なサービス構成やネットワーク等、学習すべき内容が多く、学習コストが高い。今回のWebアプリにはオーバースペックと判断し、不採用とした。 |

- 今回チャレンジしたい点
  - local Storage機能を用いたゲストユーザーでも操作ができる機能を実装したい
  - 登録制限(ある時刻によって制限解除など)をRailsのロジックで正確に実装したい
  - スムーズに画面が動くような快適なユーザ体験を提供したい

- 不安な点
  - local Storage機能やJavaScriptでの実装に関する理解が曖昧なため、実装時にキャッチアップが必要
  - 時刻によって制限を外すなどの方法は手段が多数ありそうなので、最適な方法を考える必要がある
  - ページをスムーズに動かすためのパフォーマンス向上のための知見があまりない

---

### 10-3.キャッチアップ不足の懸念

1. JavaScriptやlocal Storage等の理解が浅く、調査範囲が不明確なため時間がかかる可能性がある

    - JavaScriptやlocal Storage等のフロントで動き機能について理解が浅く、どのような仕組みかを整理し、今回の機能実装にはどういった設計で取り入れるかを考える必要がある。
    - 「何を調べれば実装できるのか」という調査範囲自体が曖昧なため、情報収集や試行錯誤に時間がかかる可能性がある。

2. Issueの工数見積もりが難しい

    - 上記1のように調査範囲が曖昧であり、理解できている部分と理解不足な部分の境界も明確ではない。
    - そのため調査時間や実装難易度を正確に見積もることが難しく、Issueごとの工数や開発期間の予測精度が下がる。

3. 設計や実装のベストプラクティスの判断が難しい

    - 時刻による制限の解除等、多数実装方法が考えられる。
    - どれが保守しやすい設計・実装かを判断できず、実装方法の比較・検討に時間を要する可能性がある。

4. パフォーマンス改善の知見が少なく、ボトルネックの特定が難しい

    - ページ表示やアニメーションを快適にするための知識が少なく、どこがボトルネックになっているかを判断するのが難しい。
    - パフォーマンス改善には様々な手法があるため、どの改善策を優先すべきか調査が必要になる可能性がある。

5. ロジックの責務設計に悩む可能性がある

    - ポイント付与・消費や利用制限などのバックエンド機能では、Controller・Model・Serviceなど、どこに処理を書くべきか判断が難しい。
    - 設計方針を誤ると、後から機能追加や保守がしづらくなる可能性がある。

#### 対応方針

- 実装に入る前に、公式ドキュメントやAIで事前調査をして、仕組みや役割を理解してから開発を進める。
- 分からない技術は、小さなサンプルを作って動きを確認しながら理解を深め、調査範囲を少しずつ明確にしていく。
- 工数を見積もる際は、実装だけでなく調査や検証にかかる時間も含めて考え、余裕を持ったスケジュールを立てる。
- 設計や実装方法に迷った場合は、まずは動くものを作ることを優先し、その後より良い設計へ改善していく。
- パフォーマンスについては、最初から最適化を目指すのではなく、明らかな問題が見つかったタイミングで原因を調べて改善する。
